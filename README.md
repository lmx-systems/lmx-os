# LMX OS — Phase 1 Core Backend + Orchestrator Dashboard

This is the first real slice of LMX OS: the backend services that take an
order from "just arrived from a shop's POS" through SLA classification,
batch-holding, and a dispatch assignment, plus an internal dashboard to
watch it happen. It follows the three-layer architecture in
`LMX_OS_Technical_Design_2.md` (Layer 2: Intelligence), built on top of
Layer 1 commodity infra (Postgres, Redis) that we own directly rather than
buying.

**What this is not:** the driver mobile app, the client-facing dashboard,
shop SMS, or a validated DPH number. Also, critically: **the API has no
authentication yet** — see `docs/ARCHITECTURE.md` before pointing anything
here at a non-local/private environment. Full scope breakdown and what's
deliberately stubbed for now is in `docs/ARCHITECTURE.md`; the live punch
list is in `docs/NEXT_STEPS.md`.

## Quickstart

```bash
cp .env.example .env          # fill in real values as they're provisioned
docker compose up --build      # postgres + redis + app (:8000) + dashboard (:5173)
docker compose exec app alembic upgrade head
```

Health check: `curl http://localhost:8000/health`

Interactive API docs: `http://localhost:8000/docs`

Dashboard: `http://localhost:5173` (paste a hub UUID to load data)

## Local development (without Docker)

Backend:

```bash
pip install -r requirements-dev.txt
# requires local Postgres + Redis, or point .env at hosted instances
alembic upgrade head
uvicorn app.main:app --reload
```

Dashboard (separate terminal):

```bash
cd dashboard
cp .env.example .env
npm install
npm run dev   # http://localhost:5173
```

## Tests

Backend:

```bash
pytest tests/ -v --cov=app --cov-report=term-missing
```

All tests run against `fakeredis` and pure functions — no live Postgres or
Redis needed. There are no integration tests against a real database yet;
see `docs/ARCHITECTURE.md` for why and what to add next.

Dashboard:

```bash
cd dashboard
npm run build   # type-check + build
npm run lint
```

## Repository layout

```
app/
  models/         Postgres schema (SQLAlchemy) - hubs, clients, shops, drivers,
                  orders, routes, stops, stop_flags, proposed_rules, active_rules
  ingestion/      Component 1: Order Ingestion Layer (POS/DMS adapters + service)
  sla/            Component 2: Dynamic SLA Engine (T1/T2/T3 classification)
  batch_queue/    Component 3: Batch-Hold Queue (clustering + hold/release logic)
  fleet_state/    Component 4: Fleet State Manager (Redis-backed driver state)
  optimizer/      Component 5: Dispatch Optimizer (Google Routes client + stub)
  learning_loop/  Component 6: Annotation and Learning Loop (pattern detection -> proposed_rules)
  events/         In-process event bus wiring the optimizer to real triggers
  api/            Health checks, read endpoints for the dashboard, manual trigger endpoints
  main.py         FastAPI app + startup/shutdown lifecycle
migrations/       Alembic migrations (hand-written, not autogenerated)
tests/            pytest suite, all offline-runnable
dashboard/        Component 7 (partial): internal orchestrator dashboard (Vite/React/TS/Tailwind)
docs/             Architecture + handoff notes, living next-steps tracker
```

## Configuration

All tunables live in `app/config.py`, sourced from environment variables
(see `.env.example`). Nothing is hardcoded in business logic:
`BATCH_HOLD_CLUSTER_RADIUS_MILES`, `OPTIMIZER_CYCLE_BUDGET_SECONDS`,
`DASHBOARD_CORS_ORIGINS`, and all third-party credentials are env-driven.
